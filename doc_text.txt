1.Document Review
This document provides a comprehensive technical and functional overview of theForm 13 (F13) API Integrationprocess, which is a key digital workflow for port operations within the ODeX platform. Form 13 is a critical document used in Indian ports (e.g., JNPT, Mundra, Pipavav) for gate-in permissions of containers, and this documentation aims to guide technical teams, integration partners, and stakeholders through its end-to-end automation.
The manual is intended to support bothAPI consumers(such as shipping lines or third-party software vendors) andsystem implementersby detailing the process of submitting, validating, and managing Form 13 data via secure RESTful APIs.
Objectives:
To outline the structure, data flow, and system interactions required for successful Form 13 integration.
To define API request/response standards, including field-level rules, mandatory checks, and sample payloads.
To ensure consistency and error-handling mechanisms across all Form 13 submissions.
To support faster onboarding of external systems or clients integrating with ODeX.
Audience:
This document is meant for:
Backend/API Developers
Integration Teams of Shipping Lines/Agents
Business Analysts and Project Managers
QA Teams involved in testing integration
ODeX Internal IT & Support teams
2. Business Use Case
Form 13 (F13) is amandatory document issued by shipping linesto authorize thegate-in of export containersat Indian ports such as JNPT, Mundra, and Pipavav. Traditionally handled via email or manual systems, Form 13 is now being digitized through API integration to improve turnaround time, reduce manual errors, and ensure regulatory compliance.
The Form 13 Integration enables seamless, real-time submission of container gate-inrequestsdirectly from thePayor’sinternal system to the ODeXSystem. This not only simplifiessubmission of Form 13 requestsbut also enhances operational efficiency, especially in high-volume export scenarios.

3. Integration Scope
Overview:
This integration enablesPayor’sto submitForm 13 datadirectly to ODeX via REST APIs.Shipping linevalidates andODeX shares the response back to the Payor.
In Scope:
Form 13 Transaction APIfor submitting container gate-in data.
Master APIsfor Vessel, POD, Terminal, etc.
Validationsfor mandatory fields and data format.
API Responsesfor success and error handling.
Retry Supportfor failed transactions.
Status Tracking(if enabled on ODeX UI or API).
Out of Scope:
Amendments of submitted Form 13.
EDI/SFTP-based submissions.
Interfaces:
Payor System→ ODeX API Gateway →Shipping LineReview
ODeX manages downstream logic and responses.
Security:
Auth via API key/code(shared securely).
HTTPS mandatory.
Available in UAT & Production environments

5.5Gate Pass - CancellationAPI
The Form 13 Cancellation API enables registered users to withdraw a previously submitted Gate Pass request for both import and export operations. This functionality is essential in scenarios where the original submission contains incorrect information, operational plans change, or the request was submitted in error or duplicate. Cancellations can be initiated prior to gate-in, provided the Form 13 is in an active state such as ‘Verified’or ‘Confirmed’.
In cases where the gate pass has already been confirmed, a structured Request for Cancellation (RFC) workflow is followed, requiring approval from the relevant Liner or platform authority. The cancellation process ensures data integrity, prevents unauthorized terminal access, and facilitates resubmission of corrected Form 13 details where necessary.

10. FAQs
Q1: What is Form 13?
A:Form 13 is a mandatory gate-in permission document issued by the shipping line to authorize the entry of export containers at Indian ports.
Q2: Can I submit Form 13 manually if the API fails?
A:Yes, Form 13 can still be submitted manually via the ODeX web interface in case of API failure or fallback.
Q3: What if the vessel or POD is not available in the master?
A:Use theVessel Master APIandPOD Master APIto fetch valid values. If missing, contact the shipping line or ODeX support to update master data.
Q4: Is hashKey mandatory in all API calls?
A:No. hashKey isoptional, butmandatory if additional encryption/security is required.
Q5: What should I do if I receive an “Incorrect Vessel” error?
A:Check that theVessel NameandVIA No.exactly match the ODeX vessel master list (case-sensitive, no extra spaces).
Q6: How can I get the list of valid attachment types?
A:Refer to theAttachment Titlessection in this document or use the code list provided by ODeX.
Q7: What happens if required fields are missing?
A:The API will return a structured error message. Review the error, fix the request, and re-submit.
Q8: Can I track the status of a Form 13 submission?
A:Yes, use theGet Status APIto retrieve the real-time status of your Form 13 requests.
Q9: Can the same API key/token be used across environments?
A: No,separate keys/tokens are shared for UAT and Production environments.
Q10: Is it possible to submit multiple containers in a single Form 13 request?A: Yes,the API supports batch submission. Each container must be structured individually within the request.Q11: Are partial failures handled during bulk container submissions?A: No,validation is applied at the container level but the whole request may be rejected if any required field is missing or invalid for any container.Q12: What is the typical response time for Form 13 API requests?A: API responses are returned in real-time, typically within 2–3 seconds under normal load.
Q13: What should I do if a port, vessel, or POD is missing from the master list?A:You should raise a request with ODeX support to have the master updated.Q14: How often are master data APIs (Vessel, POD, Terminal) updated?A:They are updated as per changes received from shipping lines. Data fetched via timestamp-based APIs ensures the latest information.
Q15: What happens if an incorrect cargo type or container size is passed?
A:The API will return an appropriate validation error. Refer to the master values shared in the documentation to correct it.
Q16: Is VGM mandatory for all containers?
A:VGM is mandatory only if"VGM via ODeX"is set to"N".If set to"Y",ODeX will fetch VGM details internally.
Q17: What file formats are accepted for attachments?A:OnlyPDF filesare accepted. The size should not exceed 4 MB per attachment.
Q18: Can the same attachment title be reused for multiple containers?A: Yes,provided each Doc details are provided in the attachment section.
Q19: Can failed API requests be retried automatically?A: Yes,retry logic can be implemented on the client side based on the error messages returned.
Q20: How are amendments to submitted Form 13 handled?A:Amendments are not supported via API. You must log into the ODeX UI to raise modification requests.
Q21: How can we track historical Form 13 requests?A:Use theGet Form 13 Status APIto retrieve current and past statuses of submitted requests.
Q22: Are there any audit logs for API transactions?A: Yes,all API transactions are logged by ODeX for audit and support purposes.
